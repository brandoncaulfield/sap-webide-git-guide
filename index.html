Important Concepts in SAP WebIDE

SAP WebIDE allows you to create a git repository for each project you work on. The creation of the repository is controlled by SAP WebIDE and if you want to work on the application you will need to clone the repository into your own git account.

Remeber, you have to commit and also push your changes to the remote repository if you want other developers to see your changes.

Git Guide: initial creation

Create local repository in WebIDE

  1. In SCP create the remote Git repository (for the required URL)
  2. Then in WebIDE we set the remote repository for the app from the step above
  3. First pull from the newly created remote repository to merge your initial files with the remote master.
  4. Then push any changes you migh'tve made or just push anyway to be safe
  5. From there another user should be able to clone the newly created remote repository

Using Branches

Scenario:

You have a master remote branch that is stored on github.

  1. Main Developer: has a local copy of the remote master (local master) that they put there originally using the set remote button/command.
  2. 2nd Developer has a local copy of the remote master (local master)

Each developer can now leave the master branch as is which means that there are effectively 3 versions of the master branch:

  1. GitHub: origin/master (remote)
  2. Main Developer: origin/master (local to their laptop)
  3. 2nd Developer: origin/master (local to their laptop)

All three versions contain the same code for now.

Then each developer decides to make their own changes and so they both create their own unique local branch of the local origin/master version they have on their local machines.

The main developer creates a local branch called: changeABC_branc

The 2nd developer creates a local branch called: changeXYZ_branch.

In our scenario we now have the following branches:

GitHub:

  1. origin/master (remote on GitHub servers)

Main Developer:

  1. origin/master (local to their laptop)
  2. changeABC_branch (local to their laptop)

2nd Developer:

  1. origin/master (local to their laptop)
  2. changeXYZ_branch(local to their laptop)

The idea with each developer creating a branch for each change is that they can make the change in this branch and then before committing to the remote repository the can pull any changes made by other developers to their local origin/master branch. Then they can try to merge the changes from, for example changeXYZ_branch, locally and test if their changes will work with the other developers recent changes.

This helps in scenarios where the main developer has mase changes yesterday and you want to merger your changes with his. In this scenario, after you've finished working on your custom branch, you simply

Change back from the changeXYZ_branch to the origin/master branch.

Then you pull/ fetch the changes from the remote origin/master branch to make sure you have the most up to date changes from the remote repository on GitHub.

Then you merge your origin/master branch with the remote origin/master by clicking on the merge button.
